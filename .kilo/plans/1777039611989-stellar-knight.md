## Implementation Strategy

### Phase 1: Unify Data Layer (Week 1)
**Objective:** Single source of truth for all fleet state

1. **Extend D1 Schema** (`migrations.js` v7)
   ```sql
   -- Worker registry
   CREATE TABLE workers (
     id TEXT PRIMARY KEY,
     name TEXT UNIQUE,
     type TEXT, -- 'command-center', 'social', 'bouncer', etc.
     version TEXT,
     deployed_at INTEGER,
     status TEXT, -- 'active', 'inactive', 'error'
     last_health INTEGER,
     config TEXT -- JSON
   );
   
   -- Deployment audit log
   CREATE TABLE deployments (
     id INTEGER PRIMARY KEY AUTOINCREMENT,
     worker_id TEXT,
     version TEXT,
     triggered_by TEXT,
     status TEXT, -- 'pending', 'success', 'failed'
     logs_url TEXT,
     created_at INTEGER
   );
   
   -- CRDT operation queue
   CREATE TABLE crdt_queue (
     id INTEGER PRIMARY KEY AUTOINCREMENT,
     operation TEXT, -- JSON
     target TEXT, -- 'mesh', 'worker', 'social'
     priority INTEGER,
     status TEXT, -- 'pending', 'processing', 'completed', 'failed'
     created_at INTEGER,
     processed_at INTEGER
   );
   ```

2. **Centralized Telemetry Aggregator** (`src/telemetry.js`)
   - Poll all worker health endpoints every 30s
   - Store in `STATUS_KV` and D1 `workers` table
   - Emit WebSocket events (via Durable Object?) for real-time UI updates

3. **Unified Auth Layer** (`src/auth.js`)
   - Extend existing Cloudflare Access JWT validation
   - Add scopes: `mesh:read`, `mesh:write`, `fleet:deploy`, `fleet:admin`, `cf:read`, `cf:admin`
   - Role-based access control (RBAC) for G.O.D. console

### Phase 3: Worker Integration (Week 3)
**Objective:** Connect all workers to G.O.D. dashboard

1. **Deploy Trigger System**

2. **Health Aggregation Service** (`src/health-aggregator.js`)
   - Runs every 30s (scheduled worker)
   - Checks: `command-center`, `bouncer`, `social-drop-automation`, external mesh services
   - Stores in D1 `workers` table
   - Pushes updates to G.O.D. UI via Server-Sent Events (SSE)

3. **CRDT Queue Processor**
   - Move from lazy mesh read to active queue
   - Workers push operations to `crdt_queue`
   - Dedicated Durable Object processes queue in order
   - Conflict resolution via vector clocks
   - UI shows queue depth and processing rate

4. **Logging Infrastructure**
   - All workers write structured logs to D1
   - Include: timestamp, worker, level, message, context
   - G.O.D. UI provides log viewer with filtering
   - Real-time tail via SSE

## Risk Mitigation

### Technical Debt
- **Risk:** Over-engineering the unified dashboard
- **Mitigation:** Build incrementally, replace components gradually
- **Fallback:** Keep existing systems operational during transition

### Performance
- **Risk:** Cloudflare API rate limits
- **Mitigation:** Aggressive caching, exponential backoff, queue requests
- **Fallback:** Serve stale data with warning indicator

### Security
- **Risk:** Exposing Cloudflare API tokens in frontend
- **Mitigation:** Backend-for-frontend pattern, never expose tokens to client
- **Fallback:** Read-only mode if auth fails
